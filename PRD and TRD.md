## **Part 1: 제품 요구사항 (PRD - Product Requirements Document)**

### **2. 개요 (Overview)**

#### **2.1. 문제 정의 (Problem Statement)**

슬랙(Slack)은 효율적인 업무 커뮤니케이션 도구이지만, 사용자는 메시지를 작성할 때 종종 어려움을 겪는다. 특히, 외부 고객이나 상급자와 소통할 때 더 전문적이고 정중한 톤으로 메시지를 다듬고 싶거나, 긴 내용을 간결하게 요약하거나, 빠르게 작성하면서 발생한 오탈자를 수정하고 싶을 때가 많다. 현재 슬랙 내에는 이러한 텍스트 후처리(Post-processing)를 직접 지원하는 기능이 없어, 사용자는 별도의 편집기 앱을 오가거나 메시지를 보낸 뒤 급하게 수정 또는 삭제하는 번거로운 과정을 반복하고 있다.

#### **2.2. 제안 해결책 (Proposed Solution)**

**AI Post**는 사용자가 슬랙에서 작성한 메시지 초안을 AI를 통해 원하는 형태로 가공하고, 그 결과를 **사용자 본인의 이름으로** 원래 보내려던 채널이나 스레드에 게시해주는 슬랙 앱이다. 사용자는 모달(Modal) 인터페이스 또는 단축 명령어(CLI)를 통해 AI 처리를 '예약'할 수 있으며, AI가 작업하는 동안 기다릴 필요 없이 다른 업무를 계속할 수 있다. 이 모든 과정은 비동기적으로 처리되어 사용자의 업무 흐름을 방해하지 않고 매끄러운 경험을 제공한다.

### **3. 사용자 페르소나 및 핵심 시나리오**

#### **3.1. 페르소나**

- **신입사원 김하나**: 고객사와의 커뮤니케이션이 잦다. 자신의 메시지가 항상 전문적이고 정중한 톤을 유지하기를 원하며, 사소한 오타로 인해 신뢰를 잃을까 걱정한다.
    
- **팀장 박준서**: 여러 프로젝트를 동시에 관리하며 팀원들에게 명확하고 간결한 지시나 공지를 전달해야 한다. 장황한 설명보다는 핵심만 요약하여 전달하는 것을 선호하며, 반복적인 작업을 명령어로 빠르게 처리하고 싶어 한다.
    

#### **3.2. 핵심 사용 시나리오 (Key Scenario)**

1. **상황**: 팀장 박준서가 팀 채널에 주간 업무 요약 공지를 올리려 한다. 그는 이미 '주간 보고 요약'이라는 이름의 커스텀 프롬프트를 만들어 두었다.
    
2. **호출 (CLI 모드)**: 박준서는 모달을 열지 않고, 메시지 입력창에 `/ai "주간 보고 요약" 이번 주 주요 진행 사항은 A, B, C이며...` 형식으로 명령어를 입력한다.
    
3. **비공개 피드백**: 앱은 즉시 박준서 팀장에게만 보이는 비공개 메시지로 `✅ '주간 보고 요약' 프롬프트 요청이 접수되었습니다.` 라고 응답한다.
    
4. **비동기 처리**: 박준서는 즉시 다른 DM 채널로 이동해 다른 업무를 본다. 그 사이, AI Post의 백그라운드 시스템은 AI API를 호출하여 장문의 텍스트를 핵심만 간추린 목록 형태로 가공한다.
    
5. **결과**: 몇 초 뒤, AI가 요약한 최종 공지사항이 **박준서 팀장의 이름과 프로필 사진으로** 팀 채널에 자연스럽게 게시된다.

## **Part 2: 기술 요구사항 (TRD - Technical Requirements Document)**

### **5. 기술 아키텍처 (Technical Architecture)**

본 프로젝트는 실시간 요청 처리와 시간이 소요되는 백그라운드 작업을 효율적으로 분리하기 위해 **3-Tier 아키텍처**를 채택한다.

- **Tier 1: Web Server (Flask App)**: 슬랙 API로부터 오는 모든 HTTP 요청(슬래시 명령어, 모달 상호작용 등)을 실시간으로 수신하는 진입점. 빠른 응답이 필수적이므로 무거운 작업은 직접 처리하지 않는다.
    
- **Tier 2: Task Queue (Celery & Redis)**: Web Server가 받은 시간 소요가 큰 작업(AI API 호출)을 등록하고 보관하는 대기열. 시스템의 부하를 분산하고 안정성을 보장한다.
    
- **Tier 3: Worker (Celery Worker)**: Task Queue를 지속적으로 감시하며, 등록된 작업을 가져와 실제로 처리하는 독립적인 프로세스. 여러 워커를 병렬로 실행하여 처리량을 늘릴 수 있다.
    

#### **데이터 흐름**

1. **요청 수신**: 사용자의 슬랙 액션이 슬랙 API를 통해 **Web Server**로 전달된다.
    
2. **작업 등록**: Web Server는 요청을 검증하고 필요한 데이터(user_token, channel_id, text 등)를 **Redis(Task Queue)**에 작업(Task)으로 등록한다.
    
3. **즉시 응답**: Web Server는 작업 등록 직후, 슬랙에 "요청 접수 완료" 메시지를 즉시 응답하여 3초 타임아웃을 피한다.
    
4. **작업 처리**: **Worker** 프로세스가 Redis에서 작업을 가져와 OpenAI API를 호출하고 결과를 기다린다.
    
5. **결과 게시**: Worker는 AI 처리 결과를 받아, 작업에 포함된 user_token을 사용하여 슬랙 API를 통해 최종 메시지를 사용자 명의로 게시한다.
    

### **6. 추천 기술 스택 (Recommended Tech Stack)**

|**Slack SDK**|**slack-sdk**|슬랙 API와의 모든 상호작용(메시지 전송, 모달 열기 등)을 간편하게 처리하는 공식 라이브러리.|

|**ORM**|**SQLAlchemy**|파이썬 코드만으로 데이터베이스 작업을 객체지향적으로 처리하게 해주는 라이브러리. SQL 쿼리를 직접 작성하는 수고를 덜어준다.|
|**AI API**|**OpenAI (Python SDK)**|GPT 모델을 활용하기 위한 공식 Python 라이브러리.|
|**배포/인프라**|**Docker, AWS/GCP**|웹 서버, Celery 워커 등 여러 구성요소를 Docker 컨테이너로 만들어 배포 일관성을 유지한다. AWS, GCP 같은 클라우드에서 안정적으로 호스팅한다.|

### **7. 데이터베이스 모델링 (Database Modeling)**

SQLAlchemy ORM을 사용하여 다음 두 개의 핵심 테이블을 설계한다.

**1. `Users` Table**

- 사용자의 인증 정보와 기본 설정을 저장한다.
    

|필드명|데이터 타입|제약 조건|설명|
|---|---|---|---|
|`id`|Integer|Primary Key, Auto-increment|고유 식별자|
|`slack_user_id`|String|Unique, Not Null|슬랙에서 제공하는 사용자 ID (예: `U024BE7LH`)|
|`slack_team_id`|String|Not Null|슬랙 워크스페이스(팀) ID (예: `T024BE7L6`)|
|`user_token`|String (Encrypted)|Not Null|**암호화 필수.** 사용자를 대신해 메시지를 보낼 OAuth 토큰|
|`created_at`|DateTime|Not Null, Default: now()|레코드 생성 시각|

**2. `Prompts` Table**

- 사용자가 직접 생성한 커스텀 프롬프트를 저장한다.
    

|필드명|데이터 타입|제약 조건|설명|
|---|---|---|---|
|`id`|Integer|Primary Key, Auto-increment|고유 식별자|
|`user_id`|Integer|Foreign Key (`Users.id`)|프롬프트 소유자. `Users` 테이블과 연결|
|`prompt_name`|String|Not Null|프롬프트의 이름 (예: "주간 보고 요약")|
|`prompt_text`|Text|Not Null|실제 AI에게 전달될 프롬프트 내용|
|`created_at`|DateTime|Not Null, Default: now()|레코드 생성 시각|

### **8. 주요 기능별 구현 전략**

- **사용자 인증 (OAuth)**: Flask에 `/slack/oauth/callback` 엔드포인트를 생성. `slack_sdk`의 `oauth.v2.access` 메소드를 사용하여 `code`를 `user_token`으로 교환 후, 암호화하여 `Users` 테이블에 저장한다.
    
- **프롬프트 관리**: `/ai-prompts` 명령어를 처리할 엔드포인트와 모달을 생성. 모달 내 버튼 클릭 시 발생하는 `block_actions` 이벤트를 수신하여 `Prompts` 테이블에 대한 CRUD 로직을 수행한다.
    
- **명령어/모달 분기**: `/ai` 엔드포인트에서 `request.form.get('text')`의 존재 여부를 확인하여 인자가 있으면 CLI 모드로, 없으면 GUI 모드로 분기. GUI 모드에서는 `client.views_open()`을 호출하여 모달을 연다.
    
- **비동기 작업 호출**: Celery Task(예: `process_ai_message.delay(...)`)를 호출하여 백그라운드 처리를 시작한다. 이때 사용자 인증, 채널 정보 등 모든 필요한 데이터를 인자로 넘겨준다.
    
- **비공개 응답**: CLI 모드 또는 모달 제출에 대한 즉각적인 응답은 `response_type: "ephemeral"`을 포함하여 HTTP 200 응답을 반환하거나, `chat.postEphemeral` API를 사용한다.
    

### **9. 개발 순서 제안 (Development Roadmap)**

1. **1단계 (MVP Core): 기본 환경 구축 및 비동기 처리**
    
    - Flask 서버 구축 및 슬랙 앱 연동.
        
    - Celery & Redis를 도입하여, `/ai [텍스트]` 입력 시 봇 명의로 AI 처리 결과를 게시하는 핵심 비동기 파이프라인 완성.
        
2. **2단계: 사용자 인증 및 권한**
    
    - PostgreSQL, SQLAlchemy 연동 및 DB 모델링.
        
    - OAuth 흐름을 구현하여 사용자 토큰을 DB에 저장하고, AI 처리 결과를 **사용자 명의로** 게시하도록 변경.
        
3. **3단계: GUI 기능 완성**
    
    - 모달 인터페이스(`/ai`) 및 프롬프트 관리(`/ai-prompts`) 기능 구현.
        
    - DB에 저장된 커스텀 프롬프트를 모달 드롭다운에 동적으로 불러오는 기능 구현.
        
4. **4단계: 고급 기능 및 안정화**
    
    - CLI 모드(`/ai "이름" "내용"`) 구현.
        
    - 에러 핸들링, 로깅, Dockerize 및 클라우드 배포 준비.
        

### **10. 성공 지표 (Success Metrics)**
